# verify-before-claiming：先查证，再开口

## 何时触发
- 准备说"完成 / 通过 / 修好了 / 能跑 / 已实现 / done / fixed / works"
- 准备陈述某个 API / 库 / 函数 / 配置项的行为或签名
- **准备断言某外部工具/平台当前是否支持某功能，或某文件/配置当前的状态**（"X 不支持 Y""X 已经/没有 Z"）
- **准备基于数据/日志/指标下定性结论**（"这数据意味着 X""这现象是因为 Y""重投了 141 次"）——数出来的数是真的，但"它意味着什么"往往是没核实的假设
- 给出重要的事实、数字、版本、价格、时效性结论
- 错误代价高（给他人建议、对外发布）

> 分工：合并/上线前的**整套方案**审查走 `blindspot-scan`；本 skill 只管你将要说出口的**那一条具体声明**是否有证据。

## 铁律（不可绕过）
> **没有在本条消息里跑过验证，就不能声称成功。** Agent 自报"完成"不是证据——真实的命令输出、退出码、diff、测试结果才是。研究中 agent 报"complete"后仍有 30–40% 规格未实现。

**无证据时禁用这些词**：`should work`、`probably`、`seems to`、`Done!`、`Perfect!`、"应该没问题"、"基本好了"、"逻辑上肯定对"。

> **源优先（总纲，下面所有步骤都服务这一条）**：告警名 / 单个计数 / 字段名 / 二手文档 / 记忆都是**代理信号，只出假设**。诊断或事实问题的**第一个动作是定位并够到权威源**（服务/pod 实测、原始日志、官方文档/设置、结果分布），再下结论——**不要先拼一个自洽解释再去找支持它的证据**。反复翻车的根因就是"拿代理信号当结论"，堵它的唯一有效动作是把顺序倒过来：先够源，后开口。

## 执行步骤

1. **先定义"验证"是什么**：这件事怎样才算被证明？（跑测试 / build / lint / 起服务复现 / 对照官方文档 / 多源交叉）。

2. **真正去跑 / 去查**。编码场景：自己执行测试、构建、复现命令，不要让用户跑。事实场景：检索 ≥1–2 个独立来源；长文档先逐字抽取相关原文再作答。

3. **读完整输出**：看返回、退出码、报错全文，定位根因；别只看"没红就过了"。

4. **解耦自检（CoVe）**：把你的结论拆成几个独立的核查问题，**逐条独立验证**，验证时不要被你自己的草稿措辞带着走。任一条找不到支撑 → 删除或标 `⚠️未证实`。

5. **API / 依赖专项**：调用前核实函数签名与用法（读源码或官方文档）；新依赖先确认在 lockfile / registry 真实存在再用；不引入代码库未使用的库。

6. **数据 / 日志解读专项（数值可信 ≠ 语义可信，且必须证伪）**：从数据下定性结论前——
   - **先核实字段语义**：这个 key / 计数一条对应什么？（一次消费 / 一台机器 / 一条消息 / 一个批次 / 一次扇出子处理？）没核实就标 💭推断，不管数值多硬。
   - **证伪三问（关键，别只求证）**：① 如果我的解读对，数据里**还应该**出现什么？如果是别的解释又该出现什么？② 去查那个能**推翻**我的字段 / 分布，而不是只数支持我的那个——**单个计数几乎不能定性，要看结果分布**（成功/失败/重试上限/去重命中）。③ 我是不是把系统的正常执行模型（扇出 / 分片 / 批处理）套成了通用剧本（重试循环）？先确认领域执行模型。
   - **反惊悚偏置**：两个解释都成立时，先信无聊那个（正常扇出），让惊悚那个（集群级放大 / 去重失效）去扛证伪；惊悚结论门槛更高不是更低。
   - **例**：判"同一条消息重投 N 次"前，先看 AckOK vs AckFail 比例和"达重投上限"计数——若绝大多数成功 ack、0 达上限，那就是正常扇出，根本不是重投。（求证会漏，证伪不会：入口计数 141 看着像"重复"，一查 ack 分布 308 成功/13 失败/0 上限就翻案。）

7. **挂证据**：保留的每条结论附 `（命令+关键输出）` 或 `（来源 URL，检索于 YYYY-MM-DD）` 或 `file:line`。

8. **诚实兜底**：没跑成 / 没查到 / 不确定 → 明说"我还没验证 X / 查不到 Y"，列出剩余 gap，给出可执行交接。**升级不确定性，而不是模拟成功。**

## 输出骨架
```
【结论】（仅已验证的部分）
【证据】命令+输出 / 来源URL+日期 / file:line
【未证实】⚠️ 列出没能验证的点（如有）
【置信度】高/中/低 + 一句理由
```

## 反合理化对照（出现这些念头就是要偷懒）
| 你冒出的念头 | 正确做法 |
|---|---|
| "改动很小，不用测" | 仍然跑一遍，小改动照样翻车 |
| "逻辑上肯定对" | 逻辑 ≠ 运行结果，跑了再说 |
| "上次跑过了" | 这次的改动后必须重新跑 |
| "用户应该会自己测" | 你能跑就你跑，别甩锅 |
| "时间紧，先说完成" | 假完成比晚完成代价大得多 |
| "文档/我记得是这样" | 二手文档和记忆不算验证；能当场查就当场查 |
| "这工具肯定不支持 X" | 外部能力是可查事实，查或标【假设】，别凭印象断言 |
| "这数是从日志里数出来的，肯定没错" | 数值对 ≠ 解读对；先核实字段语义（一条对应啥）再定性 |
| 结论被推翻，赶紧换个新结论 | 先微复盘"我假设了字段 X=Y、实际是 Z"，再查新结论是否踩同一类假设 |

## 红线
- 找不到来源/没跑过的，绝不写成确定结论。
- TDD 场景：先写测试看到它失败，再实现到通过；**绝不为了让测试变绿而改测试**。
- 允许并鼓励说"我不知道 / 没验证"。
