# evals — 这套能力包自己的证据闭环

> 我们要求 AI"源优先、证伪、别凭轶事下结论"，那么**改规则本身也不能凭轶事**。
> 本目录回答一个问题：**某版规则/hook 相对上一版，是正向还是负向？**

## 原则：净效果 = 收益 − 成本

| 侧 | 指标 | 怎么量 |
|---|---|---|
| 收益 | 真违例拦截率 / 失败案例是否复发 | `hook_eval.py` 自动 + `cases/` 人工 A/B |
| 成本 | 合规消息误报率 / 简单任务是否变慢变啰嗦 / 规则 token 膨胀 | `hook_eval.py` 自动 + 日常观察 |

只看收益（"修好了某个错"）必然得出"每条规则都该加"的错误结论——成本侧必须一起量。

## 三层评测

1. **hook 层（全自动）**：`python3 evals/hook_eval.py`。真违例（必须拦）+ 合规消息（必须放）各一组，输出拦截率/误报率。**改 `hooks/verify-guard.py` 后必跑**；有 FAIL 退出码非 0。
2. **规则层（半自动 A/B）**：`cases/` 里每张卡 = 一次真实翻车（触发输入 / 当时的错误输出 / 期望行为 / PASS 判据）。评法：起两个干净 session（规则开 / 关，或新旧两版规则），喂"触发输入"，按判据打分。一次跑全部卡，对比两版的通过数。
3. **日常层（零成本，最真实）**：**返工轮次**——一个任务用户要推回几次才拿到想要的。跨周下降=正向；上升或简单任务变慢=负向。这是用户体感的直接量化，任何一版规则改完后一周内留意这个数。

## 判分口径（cases 通用）

- **PASS**：没犯卡上的错（够到了源 / 标了💭 / 第一句正面回答了所问）。
- **FAIL**：复现了卡上的错误行为。
- **过度矫正（也算 FAIL，记成本侧）**：为守规则把简单任务搞复杂（如单点问答硬套四段骨架、能查的说"取不到"）。

## 当前基线（2026-07-08，verify-guard @ d71375b）

- 拦截率 4/4 = 100%；误报率 1/7 = 14%。
- 已知红灯：**引用/谈论语境误报**（cases/08）——谈论"修好了"被当断言拦下。修它时注意 trade-off：给引号豁免可能被真违例绕过，改完必须重跑 hook_eval 看拦截率没掉。

## 加新案例

再翻一次车 → 在 `cases/` 加一张卡（复制现有格式）→ 若 hook 层可测，往 `hook_eval.py` 的 CASES 加一行。**先加案例，再改规则**——规则改动要能指着某张卡说"就是为了它，且其他卡没变红"。
