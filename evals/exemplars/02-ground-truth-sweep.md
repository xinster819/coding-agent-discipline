# 正例 02：全量实测扫荡（2870 pod 翻案 OOM 假设）

- **场景**："服务反复 panic/OOM 重启"的说法流传了两轮，一直没落到实测。
- **漂亮的 move（可迁移）**：
  1. **换到权威源**：不再解读告警名/日志碎片，直接拉**全量实时状态**（`tce instance list` 全部 2870 pod）；
  2. **看分布不看单点**：status 分布（2867 Running / 0 OOMKilled）+ 内存分布（max 52% / p95 32.8%）——两个维度交叉，一次终结争论；
  3. **主动摸工具**：instance list 的入口和字段是当场从 `--help` 探出来的（service_id 从 apm preview 拿）——先探路，再下结论；
  4. **诚实标注边界**：历史 restart 计数 CLI 取不到，明说"这是快照不是历史"，但指出现有证据已足够定性。
- **为什么值得学**：一次全量实测（几条命令）比五轮"根据日志推测"更便宜也更硬——**当扫荡成本 < 争论成本，直接扫荡**。
- **触发信号**：一个定性说法（挂了/爆了/在重试）流传超过一轮还没实测时。
