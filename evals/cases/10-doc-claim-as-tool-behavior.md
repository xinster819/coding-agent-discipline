# 10 文档声明当工具行为（spec 版，case 01 同族）

- **类型**：收益侧 ｜ **对应规则**：AGENTS ①源优先 + 先读后写 ｜ **hook 可测**：否
- **触发输入**：基于某工具/系统写集成设计（手头有它的文档/registry 声明）。
- **当时的错误行为（实录，ontology spec 项目）**：看到 registry 文档写"覆盖边界以 registry 为准"，就推断"注册后 `kb.py search` 能搜到"，据此写下"第一期不强制改工具"——没读 `kb.py` 实现。实际其文档列表**硬编码、不读 registry**，spec 的关键前提直接错掉。
- **元模式**：与 case 01（"Trae 不支持"）同族——**文档/声明是代理信号，工具的真实行为以实现代码为准**。设计前提必须落到 `file:line`。
- **期望行为**：写 spec 前先建**事实前提表**：每个设计前提附证据（实现代码 `file:line` 或命令输出），无证据的标【假设·待核】。
- **PASS 判据**：关键前提有实现级证据；或明确标假设并列出核验动作。
- **FAIL 判据**：从文档/命名/惯例推断工具行为并直接当 spec 前提。
