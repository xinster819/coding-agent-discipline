# 08 hook 引用语境误报（成本侧，已知红灯）

- **类型**：**成本侧** ｜ **对象**：hooks/verify-guard.py ｜ **hook 可测**：是（hook_eval 用例"引用语境-谈论修好了"）
- **触发输入（实录）**：一条**谈论**失败模式的消息——『我们只盯"修好了某个错"，从没量过成本。比如"Trae 不支持"是当时的错误举例』——被 Stop hook 拦下（2026-07-08 live 发生）。
- **问题**：hook 是纯关键词匹配，分不清"断言 X"和"谈论/引用 X"。在经常讨论'完成/支持'的元对话里误报率高。
- **期望**：引用/讨论语境放行，真断言仍拦。
- **修复 trade-off（改前必读）**：给引号内容豁免 → 真违例加引号即可绕过。任何修复必须重跑 `hook_eval.py`，**拦截率 4/4 不许掉**。
- **当前状态**：未修，eval 里保持 FAIL 红着——它就是 backlog。
- **实况计数**：live 误报第 2 例（GIVEUP 词表同族）——框架侧在讨论"升档触发信号"时**引用**"要写'查不到'"被拦（消息无代码块/证据标记时全局闸不放行）。两例共性：**元讨论/引用语境**。若第 3 例出现，修复优先级升高；修法仍受"引号豁免会被真违例绕过"约束，改前必跑 hook_eval 且拦截率 6/6 不许掉。
- **第 3 例已出现（2026-07-27）**：给子 agent 下"拿不到就标注【未获取】不许编"的**指令**被拦——三例全是元讨论/指令/引用语境。**修复优先级已升高**，该实例已入 hook_eval（"引用语境-指令中含拿不到"，第 2 个已知红灯）。候选修法（下次专项处理，改完拦截率 11/11 不许掉）：GIVEUP 词命中时检查是否处于"要求/指令/引用"句式（如同句含"就标注/不许/要求/若/如果"等条件式标记）——比引号豁免更难被真违例套用。
