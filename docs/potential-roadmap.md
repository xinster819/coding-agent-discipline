# Agent 潜力工程 —— 完整方案（canonical）

> **Mission：探索如何最大化 agent 的潜力。** 规则注入只是其中一层（防下限的那层，已冻结）。
> 两个哲学（本仓库实证）：
> 1. **潜力是被"解锁"的，不是被"激发"的**——agent 表现最好的时刻（3 grep 翻案 R&F、一条命令终结 OOM 争论）全部发生在验证成本被基建压低之后。**探索半径 ∝ 1 / 行动摩擦。**
> 2. **下限与上限是两种工程**：下限靠约束（可信度地基——不撒谎/不早弃/不降级），上限靠解锁（能力基线/环境/知识/反馈/任务设计/协作）。只做下限会过拟合错误空间；只做上限会不可信。

## 一、全景：六层杠杆 + 一个地基

| 层 | 原理 | 已建资产（都在哪） | 状态 |
|---|---|---|---|
| **L1 能力基线** | 模型/effort 是硬上限，规则不可替代 | 「effort 选档」框架（见下） | ✅ 框架入宪法七（深度自调节），待数据校准 |
| **L2 环境工程** | 探索半径∝1/摩擦：降低行动成本=放大探索 | KB 工作区（biz-ontology-scaffold + content_workflow 实例，20 repo 统一搜索面）；infra-playbook（任务路由/陷阱/权限地址簿，`~/.ai-coding-pack/infra-playbook/`） | ✅ 最强的一层 |
| **L3 知识复利** | 每次踩坑/踩通→永久资产，跨 session/项目共享 | question_log（问答台账）、playbook 回写纪律、长期记忆、CHANGELOG、handoff 三件套 | 🟡 机制齐，靠纪律驱动 |
| **L4 反馈回路** | 知道什么是好，才能变好；一切争论用数据终结 | evals/：hook 自动评测（拦截 6/6、误报实时计数）＋14 失败卡＋**3 正例卡（exemplars）**；KB 抽检+熔断制度；**三指标台账 `evals/metrics-log.md`（2026-07-26 起记）** | 🟡 台账已启用；抽检待业务侧首跑 |
| **L5 任务设计** | 同一 agent，任务表述不同表现天差地别 | goal/Stop-hook（实证有效）；覆盖矩阵；验收先红后绿；**高预算任务简报模板**（`docs/task-briefing.md` + **`/task-brief` 命令**） | ✅ 一条命令可用 |
| **L6 协作结构** | 单 context 的盲区靠另一个 context 补 | fresh-context 反驳（SOP/抽检流程内）；**`/cross-check` 命令**（隔离上下文+四把刀+三态产出） | ✅ **已首跑并见效**：首轮审查发现 hook 5 个逃逸漏洞并当天修复（exemplars/04） |
| **L0 下限地基** | 可信度：不撒谎/不早弃/不降级/不虚标 | 宪法七章、10 skill、5 command、verify-guard hook（全局已配）、trae bundle | ✅ **冻结**：只减不加，只攒正例 |

## 二、L1：effort 选档（科学选，不盲目最大化）

> "默认拉满"与"凑 loop 次数"同族——把投入当效果的代理。

**第 0 步**：能不用算力就不用——playbook / KB / 正例能答的，别用推理答（知识免费，算力不免费）。

**基础档两轴定**（错误代价 × 验证成本）：

| | 验证便宜（有测试/doctor/能实测） | 验证昂贵（用户无法核验/一次性决策） |
|---|---|---|
| **错误代价高** | 中档 + 快迭代（机器验证兜底） | **高档 + 双证据链**（错误直接变用户的错误决策） |
| **错误代价低** | 低档 | 中档 |

**修正**：推理密集（因果链/设计权衡/证伪）升半档；机械活（转格式/查表/执行）降到底。
**触发式升档（代替预设拉满）**：默认中档起步，信号出现当场升档重想——卡住两次、结论反直觉、两解释并存、准备下高风险断言、准备写"查不到/无遗留"。
**数据校准**：某类任务低档反复出错→该类默认升档；高档没比中档好→降回（纳入 L4 观测）。

## 三、L2+L3：环境与知识（解锁探索的主引擎）

- **KB 工作区**（每个业务一个）：统一 search/doctor 入口、answer_contracts（问题→最终结果字段）、可决策性标注、覆盖矩阵。新业务 `bash init.sh <目录> <域名>` 一条命令实例化。
- **infra-playbook**（跨业务共享，setup.sh 永不覆盖）：tool-routes（任务→已验证命令）/ pitfalls（陷阱）/ access-directory（权限三件套，**禁编内部 URL**）。
- **回写纪律（复利的来源）**：踩通新入口→当场回写 playbook；答过的问题→question_log；发现→CHANGELOG。**不回写=白踩。**
- 使用顺序：先查手册 → 没有才枚举摸索 → 摸通必回写。

## 四、L4：反馈回路（用数据代替争论）

- **正负并重**：翻车→失败卡（14 张，先加案例再改任何东西）；干得漂亮→**正例卡**（好的探索长什么样：证伪闭环/全量扫荡/入口枚举连环推进）。
- **hook 评测**：改 hook/规则必跑 `python3 evals/hook_eval.py`（拦截率不许掉，误报实时计数，第 3 例引用语境误报出现即修）。
- **KB 校准**：每周期抽检 ≥3 条已交付答案（fresh context 独立证伪），错误率进 CHANGELOG；`[可直接决策]` 被推翻→全库标签熔断。
- **两周三指标**（回答"这套有没有效"）：用户纠正次数 / 同类错误复发数（对照案例卡）/ 新变体首现数。①②降=有效；全不降=换方向。

## 五、L5：任务设计（用户侧最便宜的杠杆）

派高预算/高价值任务用 `docs/task-briefing.md` 模板：**目标（结果语言）+ 完成判据（先红后绿）+ 覆盖要求（环节×数据源矩阵先交）+ 预算（上限不是 KPI）+ 验收方式**。Claude Code 里把完成判据设为 `/goal` 停止条件——"可以 loop 100 次"写在 prompt 里没用，写成**停止条件**才有用（实证）。

## 六、L6：协作结构（下一步落地）

高风险结论（要进报告/驱动决策的）交付前，由**不带原上下文的独立 agent** 走一遍证伪（Claude Code 用 subagent；其他工具开新 session 喂结论+原始数据）。抽检（L4）即用此结构，两件事一并落地。

## 七、运行机制（对维护者=我自己）

1. **遇新 bad case 第一问不是"加什么规则"**，而是：哪一层的问题？（能力→L1；摩擦→L2；踩过没记→L3；不知道什么是好→L4；目标没说清→L5；单 context 盲区→L6）该层杠杆用了吗？
2. **规则层冻结**：只减不加、只攒正例；除非案例证明是现有七章盖不住的下限漏洞。
3. **一切改动可追责**：先立案例卡（实录向一手记录核准）→ 指卡改动 → 跑评测 → 其他卡不许变红。

## 八、用户操作面（你的动作，全部一句话级）

- **一次性**：`bash setup.sh`（分发全部资产）；确认 hook 全局配置（已完成）。
- **每个高价值任务**：用简报模板派活；Claude Code 加 `/goal`。
- **每周**：扫 KB CHANGELOG（含抽检错误率）；看到 agent 干得漂亮/翻车，丢给我立正例/失败卡。
- **两周后**：看三指标，决定这套体系是继续、调整还是换方向。

## 九、诚实边界

- 创造力无法被规则注入——只能被允许（宪法七·探索精神）、被示范（正例）、被基线决定（L1）。
- 规则从失败归纳，天然滞后一步：**防不住新变体的首次出现**，能压复发、能让错误更早被抓。
- 本方案自身也受 L4 约束：两周数据不支持，就按数据改方向，不恋战。
