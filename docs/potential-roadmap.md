# Agent 潜力工程 —— 六层杠杆全景与路线

> Mission：本仓库的目标不是"规则注入"，是**探索如何最大化 agent 的潜力**。
> 规则只是六层杠杆之一（而且是防下限的那层）。潜力 = 能力基线 × 环境 × 知识复利 × 反馈 × 任务设计 × 协作结构。
> 核心洞察（本仓库实证）：**潜力是被"解锁"的，不是被"激发"的**——agent 表现最好的时刻（3 grep 翻案、一条命令终结 OOM 争论）都发生在"验证成本被基建压低"之后。探索半径 ∝ 1 / 行动摩擦。

| # | 层 | 管什么 | 现状 | 下一步 |
|---|---|---|---|---|
| 1 | **能力基线**（模型/effort/上下文） | 潜力的硬上限，规则不可替代 | ❌ 未管理，靠临场手选 | 任务类型→档位约定：高风险取证/整理/设计 = 最强模型 + 最高 effort；机械活可降档。**需用户拍板成默认约定** |
| 2 | **环境工程**（降低行动摩擦） | 探索半径的放大器 | ✅ 最强的一层：KB 统一搜索面、infra-playbook、tce/rds 已验证通道 | playbook 持续长厚；每类资源都有"已验证取数命令"；新工具接入即录 |
| 3 | **知识复利**（跨 session 回写） | 让每次踩坑/踩通变成永久资产 | 🟡 机制齐（question_log/playbook/记忆），靠纪律驱动、有漏记 | 把回写做成零摩擦（收工钩子里带一问："这次踩通了什么，回写了吗"） |
| 4 | **反馈回路**（eval/正负例/抽检） | 知道什么是好，才能变好 | 🟡 hook eval 自动化✅；正例库刚起步（3 张）；**抽检 0 次执行**；规则层 A/B 0 次 | 两周三指标（纠正次数/同类复发/新变体）拿基线；KB 抽检真跑第一轮；正例持续攒 |
| 5 | **任务设计**（用户侧杠杆） | 同一 agent，任务表述不同表现天差地别 | 🟡 goal-hook 验证过一次有效；覆盖矩阵/验收先红后绿已入规则 | **高预算任务简报模板**：目标写成停止条件 + 完成判据 + 显式预算 + 验收方式——"loop 100 次"写在 prompt 里没用，写成停止条件才有用 |
| 6 | **协作结构**（multi-agent） | 一个 context 的盲区靠另一个补 | ❌ 仅在 SOP 里提了 fresh-context 反驳，未落地 | 高风险结论上线前双 agent 交叉（Claude Code subagent 可直接做）；抽检即用独立 context |
| — | 规则/纪律（下限） | 可信度地基 | ✅ 已足量，**冻结**（宪法七章+10 skill+hook） | 不再加限制性规则；只做减法审计与正例 |

## 行为准则（对维护本仓库的我自己）

遇到新 bad case，**第一问不再是"加什么规则"**，而是：
1. 这是哪一层的问题？（能力不够→1；摩擦太高→2；踩过没记→3；不知道什么是好→4；目标没说清→5；单 context 盲区→6）
2. 该层已有的杠杆用了吗？
3. 规则层已冻结——除非案例证明是"下限漏洞"且现有七章覆盖不了。

## 当前优先级

1. **第 4 层拿数据**（已承诺的两周三指标 + KB 抽检第一轮）——一切"有没有效"的争论用数据终结。
2. **第 5 层做模板**（高预算任务简报）——用户侧成本最低、见效最快的杠杆。
3. **第 1 层等拍板**——模型/effort 档位约定，用户一句话的事，杠杆最大。
4. 第 2/3 层持续自然生长；第 6 层在抽检里顺带落地。
